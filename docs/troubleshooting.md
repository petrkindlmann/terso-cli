# Troubleshooting

A short list of the things people actually hit. If your problem isn't here,
open an issue at [github.com/petrkindlmann/terso-cli/issues](https://github.com/petrkindlmann/terso-cli/issues).

## `npm install -g terso-cli` fails with `EACCES`

```sh
sudo npm install -g terso-cli                  # quick, not great
# or
npm config set prefix ~/.npm-global            # better
export PATH=~/.npm-global/bin:$PATH            # add to your shell rc
npm install -g terso-cli
```

The second form keeps everything in your home directory and never asks for
sudo again.

## `terso emit` says `AGENTS.md not found`

Run `terso init` to scaffold one. By default it lands in the current working
directory, which should be your project root.

```sh
cd your-project
terso init
terso emit
```

## `terso emit --check` fails in CI with exit code `1` but local emits cleanly

That's the gate working as intended ‚Äî someone hand-edited a per-agent file
without updating `AGENTS.md`. Run `terso emit` locally, see what changed, and
update `AGENTS.md` to match (or revert the per-agent edit).

If exit code is `2` instead of `1`, that's a real error (missing file, blocked
write, unknown target). The output above the exit will tell you which.

## Windows: paths look weird in CI output

GitHub Actions' Windows runner uses backslashes. terso normalizes paths in
output, but if you see mixed `/` and `\`, that's cosmetic. The files themselves
are written correctly. File an issue if you find a case where it's not.

## `terso emit` would overwrite my hand-written `CLAUDE.md`

That's the marker-based safety net working ‚Äî terso refuses to overwrite a file
that wasn't generated by terso, unless you pass `--force`.

Pick one:

- `git diff CLAUDE.md` to see what's there, fold it into `AGENTS.md`, then
  `terso emit` (recommended ‚Äî single source going forward).
- `terso emit --force` to overwrite the hand-written file (you lose its
  contents).

## `terso doctor` says my Omnus API URL is unreachable

The `doctor` API check is an Omnus connectivity test. If you're not using Omnus
(Surface B), it's safe to ignore ‚Äî `terso emit` doesn't care.

If you are using Omnus and the check fails:

- `terso auth status` ‚Äî is your token still set?
- Try `curl https://omnus.dev/healthz` to confirm it's not a network/DNS issue.
- Check `~/.terso/config.json` ‚Äî does `apiUrl` point where you expect?

## Drift between targets

You ran `terso emit`, committed the result, then later one of the files diverged.
That usually means someone edited a per-agent file directly. To re-sync:

```sh
terso emit
```

The marker comment makes re-emit safe for terso-generated files. To prevent
drift in the first place, add the CI gate in [README.md](../README.md#ci-gate).
